# 项目开发流程说明文档

## 1. 文档目的

本文档用于说明当前 `auralmusic` 项目的实际开发流程，重点覆盖 `renderer` 侧从“新增页面”到“接接口并完成联调”的完整路径。  
内容基于当前仓库现状整理，不是通用模板，目标是让项目成员按本文即可落地功能开发。

## 2. 项目架构总览

当前项目是一个基于 `Electron + electron-vite + React + TypeScript + Tailwind CSS` 的桌面应用，代码分为三层：

1. `src/main`
   主进程代码，负责创建窗口、启动本地 Music API 服务、处理 Electron 生命周期。

2. `src/preload`
   预加载层，负责通过 `contextBridge` 向渲染进程安全暴露 Electron 能力。

3. `src/renderer`
   前端渲染层，真正承担页面开发、路由组织、组件渲染、接口请求、状态管理和样式实现。

当前功能开发的主战场是 `src/renderer`。  
除非需求涉及 Electron 能力、窗口行为或本地 API 服务，否则不要把业务页面逻辑写到 `main` 或 `preload`。

## 3. 目录职责说明

### 3.1 根目录关键文件

- `package.json`
  维护项目脚本、依赖和打包命令。

- `electron.vite.config.ts`
  维护 Electron 三端的 Vite 配置和 `renderer` 侧别名。

- `components.json`
  维护 shadcn 组件生成路径。当前 `ui` 路径指向 `@/components/ui`，即 `src/renderer/components/ui`。

- `tsconfig.app.json`
  维护 `renderer` 侧 TypeScript 路径映射，例如 `@/* -> src/renderer/*`。

### 3.2 renderer 目录职责

- `src/renderer/main.tsx`
  React 应用入口，挂载 `RouterProvider` 并引入全局样式。

- `src/renderer/router`
  维护路由配置、路由类型和路由实例。

- `src/renderer/layout`
  维护页面布局容器。当前统一走 `AppLayout`。

- `src/renderer/pages`
  页面目录。每个页面独立成一个功能目录，例如 `Home`、`Charts`、`PlayList`。

- `src/renderer/components`
  跨页面复用的业务组件，例如 `Header`、`NavBar`、`CoverCard`、`ScrollToTopButton`。

- `src/renderer/components/ui`
  基础 UI 原子组件与 shadcn 产物，例如 `button`、`card`、`skeleton`、`sonner`。

- `src/renderer/api`
  接口封装层。页面不直接写 axios，统一通过这里组织业务 API。

- `src/renderer/lib`
  公共工具和请求实例。当前 `request.ts` 是 axios 统一入口。

- `src/renderer/store`
  全局状态。当前使用 `zustand`。

- `src/renderer/hooks`
  复用 hook。当前已有分页加载 hook：`useLoadMore.ts`。

- `src/renderer/styles`
  全局样式和 Tailwind 样式入口。

## 4. 应用启动与运行链路

当前应用启动链路如下：

1. Electron 从 `src/main/index.ts` 启动主进程。
2. 主进程创建窗口，并通过 `startMusicApi()` 启动本地 Netease Music API 服务。
3. 预加载脚本 `src/preload/index.ts` 通过 `window.electron` 向渲染进程暴露基础 Electron 信息。
4. `src/renderer/main.tsx` 挂载 React 应用。
5. 路由通过 `src/renderer/router/index.tsx` 和 `src/renderer/router/router.config.tsx` 生效。
6. 页面统一被 `src/renderer/layout/AppLayout.tsx` 包裹。

结论：

- 页面开发默认只关注 `renderer`。
- 只有当页面需要使用 Electron 特性时，才考虑同时改 `preload` 和 `main`。
- 当前接口请求默认走本地 Music API 服务，不是直接在页面里拼接独立服务地址。

## 5. 新增页面标准流程

### 5.1 创建页面目录

新增页面时，统一在 `src/renderer/pages/<Feature>/` 下创建页面目录，并至少包含：

- `index.tsx`
- `components/`
- 可选的 `*.type.ts` 或 `<feature>.type.ts`

推荐结构如下：

```text
src/renderer/pages/Artist/
  index.tsx
  artist.type.ts
  components/
    ArtistHeader.tsx
    ArtistSongList.tsx
    ArtistSkeletons.tsx
```

规则如下：

- 页面容器放在 `index.tsx`
- 页面私有展示组件放在 `components/`
- 页面私有类型放在本页目录下
- 不要把页面所有 JSX、请求、状态、类型全部堆到一个文件里

### 5.2 先拆页面容器，再拆展示组件

页面实现顺序建议如下：

1. 先写页面容器 `index.tsx`
2. 明确页面分区
3. 把每个分区拆成独立展示组件
4. 若页面有明显加载态，先补骨架组件
5. 若页面有独立数据结构，再抽类型文件

当前项目推荐的拆分边界：

- 容器组件负责请求、状态、组装数据
- 展示组件负责渲染
- 基础 UI 优先复用 `components/ui`
- 页面私有组件只服务当前页面
- 多页面复用的业务组件抽到 `src/renderer/components`

### 5.3 注册路由

页面创建后，需要在 `src/renderer/router/router.config.tsx` 中注册路由。

当前项目路由配置方式是维护 `routeMenuConfig` 数组，再由 `createHashRouter(routeMenuConfig)` 生成路由实例。

新增页面时按以下步骤处理：

1. 在 `router.config.tsx` 中引入页面组件
2. 在 `routeMenuConfig[0].children` 中添加路由项
3. 补齐 `path`、`element`、`meta.title`
4. 若页面需要显示在导航中，确保 `meta.hidden` 不为 `true`

示例结构：

```tsx
{
  path: '/artist',
  element: <Artist />,
  meta: { title: '歌手', icon: '' },
}
```

### 5.4 让页面进入统一布局

当前所有主页面都挂在 `AppLayout` 下。  
`AppLayout` 已内置：

- `Header`
- `KeepAliveRouteOutlet`
- `ScrollToTopButton`

因此新增业务页面时：

- 默认接入 `routeMenuConfig[0].children`
- 不要单独绕过 `AppLayout` 另起一套页面骨架
- 若页面不需要缓存，再单独评估 `KeepAliveRouteOutlet` 的 include 策略

## 6. 接口接入标准流程

### 6.1 先写 API 层，不在页面里直接写 axios

当前仓库约定所有请求统一走 `src/renderer/lib/request.ts` 创建的 axios 实例。  
页面内不要直接 `axios.get(...)`。

标准步骤如下：

1. 确认接口属于哪个领域
2. 在 `src/renderer/api/` 下新增或补充对应模块
3. 导出语义化函数
4. 页面只 import API 函数，不直接操作底层请求实例

示例：

- 歌单相关接口放到 `src/renderer/api/list.ts`
- 专辑相关接口放到 `src/renderer/api/album.ts`

### 6.2 保持 API 函数 thin service

当前项目的 API 层应保持轻量：

- 只做路径组织
- 只做参数透传
- 不在 API 层混入页面 loading 逻辑
- 不在 API 层拼装 JSX 所需结构

建议写法：

```ts
export function getArtistDetail(id: string) {
  return request.get(`/artist/detail?id=${id}`)
}
```

### 6.3 页面中拉取数据的标准方式

当前页面主流模式如下：

- 页面容器中维护 `useState`
- 在 `useEffect` 或 `useEffectEvent + useEffect` 中请求数据
- 请求成功后更新状态
- 请求失败后记录错误日志
- `finally` 中关闭 loading

当前三个页面可以作为参考：

- `Home`
  代表分区加载、多接口并行、局部骨架

- `Charts`
  代表先取列表，再取详情，串并结合的数据聚合页面

- `PlayList`
  代表首屏请求 + 分类筛选 + 无限加载

### 6.4 优先并行请求

当多个接口互不依赖时，优先使用 `Promise.all` 并行请求，不要串行等待。

当前推荐用法：

```ts
const [a, b] = await Promise.all([getA(), getB()])
```

适用场景：

- 首页多个区块
- 页面首屏多个卡片区域
- 多个榜单详情同时拉取

### 6.5 分区 loading 优先于整页阻塞

当前项目更适合分区加载，而不是整页阻塞。

推荐策略：

- 每个区块独立维护 loading
- 先返回的区块先渲染
- 未返回的区块继续显示骨架
- 错误只影响当前区块，不阻塞全页

这已经在 `Home` 页面中体现为：

- `featureLoading`
- `artistsLoading`
- `newSongsLoading`
- `albumsLoading`

### 6.6 列表分页优先复用 useIntersectionLoadMore

如果页面是下拉加载列表，不要重复造轮子，优先复用 `src/renderer/hooks/useLoadMore.ts` 中的 `useIntersectionLoadMore`。

当前使用方式可参考 `PlayList/components/AllPlayList/index.tsx`：

1. 页面用 `useCallback` 包装获取列表函数
2. 返回 `{ list, hasMore }`
3. 将该函数传入 `useIntersectionLoadMore`
4. 使用 `sentinelRef` 作为触底加载哨兵
5. 分类或筛选条件变化时调用 `reset()`

## 7. 组件拆分与复用规范

### 7.1 三类组件边界

当前项目开发时必须先判断组件属于哪一层：

1. `src/renderer/components/ui`
   基础 UI 原子组件、shadcn 组件、通用视觉原语。

2. `src/renderer/components`
   跨页面复用的业务组件。

3. `src/renderer/pages/<Feature>/components`
   页面私有组件。

判断标准如下：

- 只在一个页面使用，放页面私有组件目录
- 多页面共用但有业务语义，放 `src/renderer/components`
- 无业务语义、纯 UI 交互能力，放 `components/ui`

### 7.2 单一职责和容器/展示分离

组件设计遵循以下原则：

- 一个组件只做一件事
- 页面容器负责取数和状态
- 展示组件只负责渲染
- 重复布局或 loading 结构要抽离
- 页面共用类型优先抽出，不要在多个组件里手写重复结构

### 7.3 骨架屏组织方式

当前项目已存在两类骨架写法：

- 页面独立骨架文件，例如 `ChartsSkeletons.tsx`
- 页面私有骨架集合，例如 `HomeSkeletons.tsx`

推荐规范：

- 页面复杂时，为该页面单独建立骨架组件文件
- 不要把大量骨架 JSX 直接写在页面容器里
- 骨架的布局尺寸应尽量贴近真实组件，减少布局跳动
- 能复用 `components/ui/skeleton.tsx` 的场景优先复用

## 8. 状态管理与数据组织规范

### 8.1 局部状态优先

当前项目大多数页面状态都在页面内部用 `useState` 维护。  
如果状态只服务当前页面，不要急着抽全局 store。

适合页面局部状态的内容：

- loading
- 当前列表数据
- 当前筛选条件
- 当前弹层显隐

### 8.2 跨组件共享状态再使用 zustand

当前全局 store 示例是 `src/renderer/store/useDailySongs.ts`。  
只有当状态需要跨页面或跨深层组件共享时，再考虑放到 zustand。

适合全局 store 的内容：

- 当前播放列表
- 全局用户信息
- 跨页面共享的推荐数据

不适合全局 store 的内容：

- 单个页面私有 loading
- 只在当前页面使用的一次性筛选状态

### 8.3 类型先行

当前项目虽然仍有部分历史代码未完全类型化，但新增功能建议坚持：

- 页面数据结构先定义类型
- API 返回结果至少在页面消费层做一次类型约束
- 组件 props 必须有明确类型

建议位置：

- 页面专属类型放在 `src/renderer/pages/<Feature>/<feature>.type.ts`
- 跨页面共享类型再考虑抽公共目录

## 9. 样式与 shadcn 使用约定

### 9.1 样式体系

当前样式体系为：

- Tailwind CSS
- `src/renderer/styles/globals.css`
- `src/renderer/styles/index.css`
- shadcn 生成的 `components/ui`

页面开发时：

- 优先复用已有 UI 原子组件
- 共性视觉能力优先进入 `components/ui`
- 页面布局和业务样式写在页面组件本身

### 9.2 shadcn 组件安装规则

当前 shadcn 配置由根目录 `components.json` 决定。  
该文件当前已配置为：

- `ui: "@/components/ui"`
- `components: "@/components"`
- `utils: "@/lib/utils"`

这意味着新安装的 shadcn 组件应落到：

- `src/renderer/components/ui`

注意事项：

- shadcn CLI 不读取 `electron.vite.config.ts` 作为主配置来源
- 组件生成路径以 `components.json` 为准
- 如果生成目录异常，优先检查 `components.json`，再检查 tsconfig 路径配置

### 9.3 Tailwind 和组件书写建议

- 优先使用 `cn()` 组织条件类名
- 不要在页面里重复写完全相同的视觉结构
- 基础按钮、卡片、骨架等优先复用已有 `ui` 组件
- 长类名建议配合 Prettier 和 Tailwind 插件整理

## 10. 联调、验证与提交流程

### 10.1 常用命令

当前根目录脚本如下：

- `pnpm dev`
  启动 Electron 开发环境

- `pnpm dev:web`
  只启动 Vite Web 环境

- `pnpm lint`
  运行 ESLint

- `pnpm build`
  执行 `tsc -b && electron-vite build`

- `pnpm check`
  执行 `pnpm lint && pnpm build`

### 10.2 新功能开发建议流程

推荐开发顺序：

1. 先梳理页面分区和数据来源
2. 创建页面目录和路由
3. 先搭页面骨架结构
4. 再补 API 层函数
5. 页面容器接状态和请求
6. 给主要区块补 loading / skeleton
7. 完成后运行 `pnpm lint`
8. 本地联调用 `pnpm dev`
9. 若仓库类型问题已清理，再跑 `pnpm build`

### 10.3 提交前检查

提交前至少确认以下内容：

- 页面目录结构正确
- 路由已注册
- API 已抽到 `src/renderer/api`
- 没有在页面里直接写 axios
- 组件职责边界清晰
- loading 与骨架齐全
- `pnpm lint` 已执行
- 需要时已在 Electron 环境中联调

## 11. 当前项目已知问题与注意事项

### 11.1 当前 build 不是全绿

截至当前仓库状态：

- `pnpm lint` 可以执行完成，但仍存在 7 个 warning
- `pnpm build` 失败

当前 `pnpm build` 的主要阻塞点包括：

- `src/renderer/api/list.ts`
- `src/renderer/components/CoverCard/index.tsx`
- `src/renderer/components/Header/index.tsx`
- `src/renderer/lib/request.ts`
- `src/renderer/pages/PlayList/components/AllPlayList/CategoriesPanel.tsx`

因此当前开发流程的现实约束是：

- 功能联调优先使用 `pnpm dev`
- 质量检查先跑 `pnpm lint`
- 完整构建验证需要先清理仓库既有 TypeScript 严格模式错误

### 11.2 当前类型规范存在历史债务

项目已经启用严格 TypeScript 配置，但部分旧文件仍存在：

- `implicit any`
- 未使用变量
- props 类型缺失
- `window.electron` 类型声明缺失

新增功能时不要沿用这些旧问题，应坚持：

- 新增组件 props 明确类型
- 新增 API 参数明确类型
- 页面状态优先声明类型

### 11.3 编码和目录一致性

当前仓库部分中文注释和文案存在历史编码问题。  
新增文档和新增代码应尽量保持编码一致、命名清晰、路径稳定，不要继续放大历史噪音。

## 12. 推荐开发范式总结

用一句话概括当前项目的正确开发路径：

先在 `renderer/pages` 下建立页面容器和私有组件，再在 `router.config.tsx` 中注册路由；接口统一封装到 `renderer/api`，页面通过 `request` 实例间接取数；页面容器负责 loading、状态和数据组装，展示组件专注渲染；列表分页优先复用 `useIntersectionLoadMore`，基础 UI 优先复用 `components/ui` 与 shadcn 组件；最后通过 `pnpm dev` 联调、`pnpm lint` 检查，并在清理完既有类型问题后执行 `pnpm build`。
